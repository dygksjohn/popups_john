# 혼잡도 예측 발표 예상 질문·답변 (2026-03-30)

## 1. 혼잡도 예측을 도입하기 전 가장 큰 운영 문제는 무엇이었나요?
행사 현장의 혼잡 피크를 사전에 못 잡아서 대기열 폭증 시점에 대응이 늦었습니다. 운영자 관점에서는 인력 재배치와 안내 타이밍이 모두 사후 대응이었습니다.

## 2. “혼잡도”를 수식으로 정확히 어떻게 정의했나요?
기본 개념은 `수요(대기/유입)` 대비 `처리능력(throughput, 수용량)` 비율입니다. 서비스 내부에서는 다수 신호를 합성해 0~100 점수로 표준화합니다.

## 3. 혼잡도 점수와 실제 대기시간의 상관관계는 어떻게 검증했나요?
예측 점수에서 대기시간을 직접 환산하는 보정 함수를 두고, 타임라인 평균 대기시간과 현장 관측 대기시간의 추세 일치 여부를 검증했습니다.

## 4. 84개 최종 입력 데이터는 어떤 기준으로 선정했나요?
최근 60개 점수 시퀀스 + 운영 파생 18개 + 시간 맥락 6개 구조로, 실시간 추론 속도와 예측 안정성을 함께 만족하는 최소 구성을 선택했습니다.

## 5. 운영 데이터 18개는 각각 무엇이고 중요도는 어떻게 되나요?
평균/표준편차/최근구간 평균·분산/분위수/추세 기울기 등 시계열 통계 파생값입니다. 중요도는 모델 학습 결과에 따라 달라지며, 최근 변동성과 추세 계열이 피크 구간에서 특히 유효했습니다.

## 6. 시간 맥락 6개가 실제 예측 성능에 얼마나 기여하나요?
요일 주기, 주말/공휴일, 행사 진행률을 반영해 같은 입력 시퀀스라도 날짜·구간에 맞는 보정이 가능해집니다. 특히 계획 행사(사전 예측) 구간에서 기여도가 큽니다.

## 7. 입력 시퀀스를 왜 60개로 정했나요?
과도하게 길면 지연이 증가하고 짧으면 패턴 정보가 부족합니다. 60개는 반응성과 안정성의 균형점으로 채택했습니다.

## 8. 데이터 수집 주기 15초, 예측 주기 5분을 선택한 근거는 무엇인가요?
15초 수집으로 급격한 변화를 놓치지 않고, 5분 예측으로 시스템 부담과 운영 활용성을 맞췄습니다. 대시보드 해석 단위로도 5분이 실무 친화적이었습니다.

## 9. 스케줄러 지연/실패 시 예측 파이프라인은 어떻게 복구되나요?
예측 API는 가능한 최신 입력으로 계산하고, 모델 불가 시 폴백 경로로 응답을 유지합니다. 로그 실패와 추론 실패를 분리해 전체 서비스 중단을 막습니다.

## 10. 대시보드에는 평균/피크/타임라인 중 무엇을 우선 노출하나요?
운영 즉시성 기준으로 피크 위험 신호를 우선 보고, 의사결정에는 타임라인과 평균을 함께 봅니다. 즉 “경보는 피크, 계획은 타임라인” 원칙입니다.

## 11. LightGBM+LSTM 조합이 단일 모델보다 실제로 얼마나 좋아졌나요?
정량 수치는 실험 리포트 기준으로 제시하고, 구조적으로는 LightGBM의 빠른 1차 예측에 LSTM 시계열 보정을 더해 급변 구간 안정성이 개선됩니다.

## 12. RandomForest/XGBoost를 탈락시킨 핵심 정량 근거는 무엇인가요?
RandomForest는 시계열 민감도 한계가 있었고, XGBoost는 동일 조건에서 학습·추론 속도 측면의 이점이 작았습니다. 운영 요구(주기 추론) 기준으로 LightGBM이 유리했습니다.

## 13. LSTM은 “보정”만 하고 주 예측은 LightGBM이 담당하는 이유가 뭔가요?
실시간 운영에서는 추론 속도와 안정성이 최우선이라 1차는 LightGBM이 맡고, LSTM은 시계열 패턴만 보강하는 역할로 분리했습니다.

## 14. Score Calibration과 Context Calibration의 역할 차이는 무엇인가요?
Score Calibration은 모델 출력의 수치 편향을 조정하고, Context Calibration은 행사 상태(진행/계획/종료), 시간 맥락을 반영해 상황 적합도를 올립니다.

## 15. Local Event Adaptor는 어떤 조건에서 켜지고 꺼지나요?
진행 중 행사이고 입력 시퀀스 길이·변동성 조건이 충족될 때만 활성화됩니다. 조건 미달이면 기본 경로로 안전하게 동작합니다.

## 16. Local Event Adaptor 재학습 주기와 트리거 기준은 무엇인가요?
시간 간격 기준(예: 재학습 인터벌)과 시그니처 변화량(평균/표준편차/최근값 이동량)을 함께 봅니다. 변화가 크면 재학습을 앞당깁니다.

## 17. Local 보정이 과적합을 유발하지 않도록 어떤 장치를 뒀나요?
최소 샘플 수·최소 변동성 조건을 두고, 글로벌 결과와 가중 블렌딩합니다. 단독 치환이 아니라 혼합 구조라 과적합 리스크를 낮춥니다.

## 18. 예측 신뢰도(confidence)는 어떻게 계산하나요?
기본적으로 최근 변동성 등 데이터 특성에 따라 범위를 제한해 산출합니다. 로컬 어댑터 사용 시에는 학습 데이터량 기반 신뢰도를 추가 반영합니다.

## 19. 예측 결과의 level(1~5) 기준은 어떻게 정했나요?
0~100 점수를 임계 구간으로 나눠 5단계로 변환합니다. 운영 화면에서 빠르게 위험도를 인지하기 위한 단순·일관 기준입니다.

## 20. 타임라인 점수는 시간대별로 어떤 규칙으로 변형되나요?
기본 점수에 시간대 프로파일, 파동(wave), 추세(trend)를 동일 규칙으로 적용해 포인트별 점수를 생성합니다. 각 포인트에서 level과 대기시간도 함께 계산됩니다.

## 21. 현재 혼잡도 계산과 미래 혼잡도 예측의 경계는 어떻게 처리하나요?
기준 시각(baseTime)을 중심으로 현재 상태 신호와 미래 타임포인트를 구분합니다. 프로그램 예측은 `max(baseTime, programStart)`부터 horizon을 잡습니다.

## 22. 프로그램 단위 예측과 행사 단위 예측 로직 차이는 무엇인가요?
행사 단위는 전체 유입·대기·운영 신호를 폭넓게 쓰고, 프로그램 단위는 체크인/대기열/처리량 중심으로 계산합니다.

## 23. 대기시간 기본 계산식과 보조 계산식은 어떤 상황에서 전환되나요?
기본은 대기열/처리량 기반이고, 신호 품질이 낮거나 특정 입력이 부족하면 anchor 기반 보조 계산과 점수 기반 환산을 혼합합니다.

## 24. Throughput 값이 누락되거나 0일 때 실제 방어 로직은 어떻게 동작하나요?
분모가 0이 되지 않도록 최소값 보정을 적용합니다(`max(throughput, 1.0)`). 따라서 계산 불능 대신 보수적 추정값을 반환합니다.

## 25. Cold-Start 상황에서 폴백 점수는 어떤 입력으로 계산하나요?
대기·신청·수용·운영 신호 기반의 휴리스틱 점수로 계산합니다. 모델 추론 불가여도 타임라인과 대기시간을 지속 제공합니다.

## 26. 폴백 사용 여부를 클라이언트/운영자가 확인할 수 있나요?
응답 필드에 `fallbackUsed`를 포함해 확인 가능합니다. 운영 모니터링에서 폴백 비율을 별도 지표로 관리할 수 있습니다.

## 27. “서비스 가용성 100%”는 어떤 지표 기준으로 산출한 값인가요?
발표 시점에서는 “예측 응답 자체가 끊기지 않음”을 의미합니다. 정확한 수치는 기간, 샘플 수, 실패 정의를 함께 제시해야 신뢰도가 올라갑니다.

## 28. “Score Jump 85% 감소”의 측정 정의(분모/기간/환경)는 무엇인가요?
반드시 실험 로그의 정의를 명시해야 합니다(급변 임계치, 비교 기간, 데이터셋). 리허설 답변에서는 “내부 테스트 기준”이라고 전제하고 수치 근거를 부록으로 제시합니다.

## 29. EMA smoothing이 모델 내부인지 후처리인지 정확히 어디에 적용되나요?
구조상 순수 EMA 단일 단계라기보다, 시계열 보정·블렌딩·클램프가 결합된 안정화 흐름입니다. 발표 문구는 “점수 안정화 보정”으로 표현하는 것이 정확합니다.

## 30. Clamp(0~100) 적용으로 생길 수 있는 정보 손실은 없나요?
극단치 정보는 잘리지만, 운영용 혼잡 점수의 도메인 범위를 유지해 해석 가능성과 안정성을 얻습니다. 이상치 폭주를 막는 안전장치입니다.

## 31. 행사별 고유 패턴(견종/활동성 등)은 어떤 피처로 반영되나요?
직접 피처와 간접 운영 신호를 통해 반영되며, 부족한 부분은 Local Event Adaptor가 행사별 잔차를 학습해 보완합니다.

## 32. 피처 드리프트(행사 시즌성 변화)는 어떻게 탐지하나요?
예측 오차 분포와 최근 시그니처 변화(평균/분산/최근값)로 감지합니다. 필요 시 모델 재학습과 로컬 보정 가중치 조정을 수행합니다.

## 33. 예측 로그는 어떤 스키마로 저장되고 재학습에 어떻게 활용되나요?
대상 타입, 기준 시각, 평균/피크 점수, 레벨, 모델 버전 등을 저장합니다. 이후 학습 데이터셋 구성이나 운영 검증 리포트 생성에 사용합니다.

## 34. 로그 저장 실패 시 사용자 응답에는 어떤 영향이 있나요?
응답 경로와 저장 경로를 분리해 저장 실패가 즉시 사용자 장애로 번지지 않게 설계합니다. 실패는 별도 로깅으로 관측합니다.

## 35. API 응답 시간(SLA) 목표와 실제 평균/최댓값은 얼마인가요?
운영 목표값은 별도 SLI/SLO로 제시해야 합니다. 발표에서는 “5분 주기 안정 추론 가능”을 말하고, 실제 p95/p99는 측정값으로 답하는 것이 맞습니다.

## 36. 동시 요청이 많을 때 모델 추론 병목은 어디서 생기나요?
주로 모델 로드·특징량 생성·I/O(로그 저장)에서 병목이 발생합니다. 캐시, 비동기 저장, 배치 처리로 완화합니다.

## 37. 장애 시 대시보드는 마지막 값 유지인지, 폴백 값 표시인지 어떻게 되나요?
원칙은 폴백 계산으로 최신 추정값을 제공하고, 불가하면 마지막 정상값 표시 전략을 병행합니다. 운영자에게 상태 플래그를 노출하는 것이 바람직합니다.

## 38. 잘못된 입력(음수, 결측, 이상치)에 대한 검증/정규화 정책은 무엇인가요?
요청 DTO 단계에서 범위를 제한하고, 내부에서는 NaN/무한대 치환 및 클램프를 적용합니다. 계산 분모는 최소값 보정으로 안정화합니다.

## 39. 공휴일/주말 맥락 정보는 어떤 캘린더 소스로 관리하나요?
고정 공휴일 테이블 + 국가 공휴일 라이브러리(가능 시) 조합입니다. 실패 시에도 기본 주말 로직으로 동작하도록 설계합니다.

## 40. 개인정보/민감정보 없이 예측이 가능한 구조인가요?
예측 입력은 운영 집계 신호 중심이라 개인 식별정보 없이 동작 가능합니다. 데이터 최소수집 원칙을 지키는 방향입니다.

## 41. 모델 버전 관리와 롤백 절차는 어떻게 설계했나요?
모델 아티팩트 파일을 버전 단위로 관리하고, 응답/로그에 모델 버전을 남깁니다. 이상 징후 시 직전 안정 버전으로 즉시 롤백합니다.

## 42. A/B 테스트나 오프라인 검증으로 성능 향상을 어떻게 입증했나요?
동일 기간 데이터에서 기준 모델 대비 MAE/RMSE, 피크 구간 오차, 급변도 지표를 비교합니다. 운영 적용 전 오프라인 검증을 우선 수행합니다.

## 43. 실제 운영 의사결정(인력 배치/동선 안내)에 어떻게 연결되나요?
피크 예측 시간대를 기준으로 인력 증원, 안내 동선 변경, 대체 프로그램 유도 등을 사전 실행합니다. 즉, 예측을 운영 액션으로 연결합니다.

## 44. 추천 프로그램 기능은 혼잡도 임계치와 어떤 방식으로 연동되나요?
현재 프로그램 점수가 임계치 이상일 때만 후보를 필터링·정렬해 대체안을 제시합니다. 임계치 미만이면 불필요한 전환을 막습니다.

## 45. 이번 구조에서 아직 남아 있는 한계와 다음 개선 계획은 무엇인가요?
정량 지표 자동 리포팅과 이벤트 특화 피처 확장이 추가 과제입니다. 다음 단계는 drift 모니터링 자동화와 피처 중요도 가시화입니다.

## 46. 행사 시작 직후/종료 직전처럼 극단 구간에서 오차 패턴은 어떤가요?
신호가 급변하거나 데이터가 희소한 구간이라 오차가 커질 수 있습니다. 그래서 폴백·컨텍스트·로컬 보정을 중첩해 안정성을 확보합니다.

## 47. 예측 실패 시 사용자에게 보여주는 메시지/UX 정책은 무엇인가요?
기본 정책은 실패를 숨기고 예측 가능한 수준의 대체값을 제공하는 것입니다. 운영 화면에는 폴백 사용 여부를 표시해 투명성을 확보합니다.

## 48. 모델·파이프라인 전체에서 가장 리스크가 큰 단일 실패 지점은 어디인가요?
입력 데이터 품질 저하가 가장 큰 리스크입니다. 모델 자체보다 데이터 지연·결측이 성능을 먼저 무너뜨리므로 수집 모니터링이 핵심입니다.
